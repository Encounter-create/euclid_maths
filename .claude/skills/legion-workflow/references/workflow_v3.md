# 罗马军团工作流 v3

> 更新日期：2026-07-03
> 基于 v2 新增 1 条法则（军令状不可改），新增快速战役模式、阵型选择、百夫长战前检查清单、归营时限机制

## 一、指挥链

```
皇帝（用户）
  │
  ├─ 宰相 AI（需求定义）   → 圣旨
  ├─ 参谋 AI（技术方案）   → 军略
  ├─ 将军 AI（任务调度）   → 军令
  │
  ├─ 前端百夫长 (file-agent)  → 代码
  ├─ 后端百夫长 (shell_executor) → 服务端代码
  ├─ QA 百夫长   (file-agent)  → QA 报告
  ├─ PM 百夫长   (file-agent)  → 文档
  ├─ UI/UX 百夫长 (file-agent) → 设计规范
  ├─ 侦察兵      (browser)     → 截图/界面验证
  └─ 宪兵        (browser)     → 自动化验收
```

## 二、启动模式

| 模式 | 适用场景 | 流程 |
|------|---------|------|
| **完整战役** | 预估 > 3 百夫长任务 | 宰相→参谋→将军 逐层转交，皇帝审阅两次 |
| **快速战役** | 预估 ≤ 3 百夫长任务 | 三合一谋划（宰相+参谋+将军合并），皇帝审阅一次 |

## 三、阵型选择

| 阵型 | 百夫长组合 | 适用项目 |
|------|-----------|---------|
| 全装阵 | 前端 + 后端 + QA + UIUX + PM | Web 全栈 |
| 轻骑阵 | 后端 + QA | CLI / 库 / API |
| 攻城阵 | 前端 + 后端 + QA + UIUX | 小程序 / 移动端 |
| 要塞阵 | 前端 + UIUX | 纯静态站点 |
| 侦查阵 | PM + (可选) 参谋 | 调研 / 方案 |

## 四、信息传递格式

| 文书 | 产出者 | 内容 |
|------|--------|------|
| **圣旨** | 宰相 | 需求定义——用户要什么，边界在哪 |
| **军略** | 参谋 | 技术方案——选什么栈，怎么拆任务，风险在哪 |
| **军令** | 将军 | 派发指令——谁去打，打哪里，打到什么程度算赢，归营时限 |
| **战报** | 百夫长 | 执行结果——打了什么，产出在哪里，自检清单通过情况，有无伤亡 |

## 五、核心法则（10条）

### 战前法则

| # | 法则 | 说明 |
|---|------|------|
| 1 | **战前谋划不可越** | 圣旨 + 军略 → 皇帝审阅 → 方可发兵。禁止跳步直接执行。 |
| 2 | **逐级抽象** | 皇帝只管"要什么"，百夫长只管"怎么实现"。上下级不越界指挥。 |

### 战中法则

| # | 法则 | 说明 |
|---|------|------|
| 3 | **百夫长继承制** | 同一战线同一百夫长，`inherit_agent_id` 保记忆，避免上下文丢失。 |
| 4 | **并行不悖** | 无数据/状态/安全依赖的任务同时派出。 |
| 5 | **增量摘要机制**（v2 新增） | 同一百夫长被继承超过 2 次时，将军在新军令中附带前两轮战报摘要。避免上下文膨胀。 |
| 6 | **军令状不可改**（v3 新增） | 发兵后皇帝追加需求，将军应将其标记为新战役而非塞进当前军令。标准话术：「陛下，此需求已超出当前军令范围，臣建议完成本轮交付后另起军令处理。」 |

### 战后法则

| # | 法则 | 说明 |
|---|------|------|
| 7 | **侦察兵先行** | browser-agent 截图为证，眼见为实。交付前必须过视觉验收。 |
| 8 | **宪兵自动化验收**（v2 新增） | 百夫长交付代码后，将军自动派 browser-agent：检查 Console 无 JS 错误 → 自动化交互验证关键路径 → 通过交付，不通过立即退回。 |
| 9 | **战损必复盘** | bug 定位 → 根因分析 → 修复 → QA 战后回归检查（v2 新增）→ 更新 qa_report.md |
| 10 | **战后必修史** | 每次战役结束撰写战报（.md）→ 存入 `battle_history/` → 提优化提案写入 `IMPROVEMENTS.md` → 皇帝批准后执行 |

## 六、百夫长归营时限机制（v3 新增）

每个军令必须包含**归营时限** = 截止时间 × 1.5。超时触发：

```
归营时限已过
     │
     ▼
斥候探查（将军主动 check）
  ├─ 收到回复 → 评估调整军令
  └─ 30min 无回复 → 判定失联
       ├─ 终止当前军令
       ├─ 保存已完成部分
       └─ 重派或拆分
```

## 七、百夫长战前检查清单（v3 新增）

每个百夫长出阵前必须执行战前自检清单 + 交付前自检清单，战报中如实汇报通过/未通过项。详见各百夫长 reference 文件：

- 前端百夫长：JS 转义/类名一致/Console 无错/响应式/测试骨架
- 后端百夫长：输入验证/SQL 参数化/环境变量/迁移可回滚
- QA 百夫长：验收标准可测/正常+边界+异常全覆盖/回归验证
- UI/UX 百夫长：色彩对比度/Token 双输出/交互状态完整
- PM 百夫长：PRD 验收标准/进度偏差监控/风险责任人

## 八、代码交付流程

```
百夫长产出代码
     │
     ▼
宪兵自动化验收（browser-agent）
  ├─ 检查 Console 无 JS 错误
  ├─ 验证关键交互路径（模式切换 / 核心按钮 / 数据流）
  ├─ 通过 → 继续
  └─ 不通过 → 退回百夫长修复
     │
     ▼
侦察兵截图验证（browser-agent）
     │
     ▼
交付皇帝审阅
```

## 九、需求变更处置流程（v3 新增）

```
皇帝追加需求（发兵后）
     │
     ▼
将军判断范围
  ├─ 小调整 → 追加令（同一军令最多 3 次追加）
  └─ 超出范围 → 法则 6 生效 → 建议另起战役 / 重走圣旨→军略→审阅
```

## 十、Bug 修复流程

```
定位根因 → 修复代码
     │
     ▼
宪兵复验（自动化验收）
     │
     ▼
QA 百夫长战后检查 → 更新 qa_report.md（修复项 + 回归结果）
     │
     ▼
交付皇帝确认
```

## 十一、工作流修改规则

1. 任何修改需皇帝明确批准
2. 修改前备份当前工作流至 `workflow_v{n}_backup.md`
3. 新版本号递增
4. 每次战后复盘是提出优化的正式窗口
5. 所有优化提案写入 `IMPROVEMENTS.md` 追踪
