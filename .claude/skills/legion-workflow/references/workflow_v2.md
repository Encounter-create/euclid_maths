# 罗马军团工作流 v2

> 更新日期：2026-06-05
> 基于 v1 新增 3 条法则（皇帝批准于首战复盘）

## 一、指挥链

```
皇帝（用户）
  │
  ├─ 宰相 AI（需求定义）   → 圣旨
  ├─ 参谋 AI（技术方案）   → 军略
  ├─ 将军 AI（任务调度）   → 军令
  │
  ├─ 前端百夫长 (file-agent)  → 代码
  ├─ QA 百夫长   (file-agent)  → QA 报告
  ├─ PM 百夫长   (file-agent)  → 文档
  ├─ 侦察兵      (browser)     → 截图/界面验证
  └─ 宪兵        (browser)     → 自动化验收（新增）
```

## 二、信息传递格式

| 文书 | 产出者 | 内容 |
|------|--------|------|
| **圣旨** | 宰相 | 需求定义——用户要什么，边界在哪 |
| **军略** | 参谋 | 技术方案——选什么栈，怎么拆任务，风险在哪 |
| **军令** | 将军 | 派发指令——谁去打，打哪里，打到什么程度算赢 |
| **战报** | 百夫长 | 执行结果——打了什么，产出在哪里，有无伤亡 |

## 三、核心法则（9条）

### 战前法则

| # | 法则 | 说明 |
|---|------|------|
| 1 | **战前谋划不可越** | 圣旨 + 军略 → 皇帝审阅 → 方可发兵。禁止跳步直接执行。 |
| 2 | **逐级抽象** | 皇帝只管"要什么"，百夫长只管"怎么实现"。上下级不越界指挥。 |

### 战中法则

| # | 法则 | 说明 |
|---|------|------|
| 3 | **百夫长继承制** | 同一战线同一百夫长，`inherit_agent_id` 保记忆，避免上下文丢失。 |
| 4 | **并行不悖** | 无数据/状态/安全依赖的任务同时派出。 |
| 5 | **增量摘要机制**（v2 新增） | 同一百夫长被继承超过 2 次时，将军在新军令中附带前两轮战报摘要。避免上下文膨胀。 |

### 战后法则

| # | 法则 | 说明 |
|---|------|------|
| 6 | **侦察兵先行** | browser-agent 截图为证，眼见为实。交付前必须过视觉验收。 |
| 7 | **宪兵自动化验收**（v2 新增） | 百夫长交付代码后，将军自动派 browser-agent：检查 Console 无 JS 错误 → 自动化交互验证关键路径 → 通过交付，不通过立即退回。 |
| 8 | **战损必复盘** | bug 定位 → 根因分析 → 修复 → **QA 战后回归检查**（v2 新增）→ 更新 qa_report.md |
| 9 | **战后必修史** | 每次战役结束撰写战报（.md）→ 存入 `battle_history/` → 提取经验 → 提优化提案 → 皇帝批准后执行 |

## 四、代码交付流程（v2 新增）

```
百夫长产出代码
     │
     ▼
宪兵自动化验收（browser-agent）
  ├─ 检查 Console 无 JS 错误
  ├─ 验证关键交互路径（模式切换 / 核心按钮 / 数据流）
  ├─ 通过 → 继续
  └─ 不通过 → 退回百夫长修复
     │
     ▼
侦察兵截图验证（browser-agent）
     │
     ▼
交付皇帝审阅
```

## 五、Bug 修复流程（v2 新增）

```
定位根因 → 修复代码
     │
     ▼
宪兵复验（自动化验收）
     │
     ▼
QA 百夫长战后检查 → 更新 qa_report.md（修复项 + 回归结果）
     │
     ▼
交付皇帝确认
```

## 六、工作流修改规则

1. 任何修改需皇帝明确批准
2. 修改前备份当前工作流至 `workflow_v{n}_backup.md`
3. 新版本号递增
4. 每次战后复盘是提出优化的正式窗口
