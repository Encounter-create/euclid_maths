# 欧几里德之门 写作交互原则

本 skill 规定了在数学公众号写作场景下，AI 与用户协作时必须遵守的三条交互原则。

## 1. 不确定即追问

当用户的指令表述不清、存在歧义，或 AI 无法准确理解用户意图时，**必须暂停并追问确认**，不得自行猜测、推断或决策。

仅在用户明确声明「自由发挥」「你看着办」等授权用语时，才可以不经追问直接执行。

## 2. 多路径必须枚举

当 AI 打算采取的行动存在多条可行路径（如不同方案、不同结构、不同选材），且无法从上下文明确判断用户偏好时，**必须将所有选项列出，让用户选择**，不得自行决策。

选项呈现要求：简洁概括每个选项的核心差异，不做冗长展开。

## 3. 先提纲后正文

写作流程中，**在撰写正文之前必须先列出提纲**。提纲应包含：

- 文章标题（或备选标题）
- 结构骨架（各板块及层级）
- 核心内容要点（每板块的关键论述/定理/例题概要，一句话即可）
- 数学史穿插的选题方向（如适用）

提纲呈现给用户，用户修改确认无误后，AI 方可进入正文写作阶段。禁止在提纲未确认的情况下直接撰写正文。

## 4. 修改前先备份

在修改任何 skill 文件（SKILL.md）或已发布的文章文件之前，**必须先将原文件备份**。

备份规则：
- 备份文件名格式：`原文件名_v{版本号}.md`，如 `SKILL_v3.md`、`SKILL_v4.md`
- 版本号从已有备份中递增（检查同目录下已有备份的最大版本号 + 1）
- 备份完成后再执行编辑，禁止直接覆盖原文
- 此规则同时适用于 skill 文件（如 SKILL.md）和已发布的公众号文章

## 适用范围

以上四条原则适用于所有触发 `euclids-gate-math-writing` skill 的写作场景。原则 1 同时适用于非写作场景。原则 2、3、4 仅在写作及相关文件操作场景下生效。
